SaaS Website vs Product Docs — What Goes Where

CORE DIFFERENCE
---------------
Website = convince and qualify
Docs     = enable and unblock


SAAS WEBSITE
------------
Audience:
- New visitors
- Prospective customers
- Decision-makers
- Evaluators skimming quickly

Primary goals:
- Communicate value
- Build trust
- Convert (signups, demos, trials)
- Set expectations

What belongs on the website:
1. Value & positioning
   - What problem you solve
   - Who it’s for (and who it’s not)
   - Key outcomes and benefits
   - Differentiation vs competitors

2. High-level feature explanations
   - What the product does
   - Concepts, not clicks
   - Short use cases or scenarios

3. Marketing & conversion
   - Landing pages
   - Calls to action
   - Pricing
   - Case studies and testimonials

4. Trust & credibility
   - Security & compliance (high level)
   - Customer logos
   - About / mission
   - Status and uptime info

5. Onboarding entry points
   - Getting started overview
   - Product tours
   - Links into docs

What should NOT be on the website:
- Step-by-step instructions
- Full configuration details
- Edge cases and troubleshooting
- Technical reference material


PRODUCT DOCS
------------
Audience:
- Active users
- Admins / developers
- Power users
- Support-seeking customers

Primary goals:
- Enable successful usage
- Reduce confusion and errors
- Support self-service
- Prevent support tickets

What belongs in the docs:
1. Step-by-step instructions
   - Setup and configuration
   - Exact UI actions
   - Screenshots and examples

2. Deep conceptual explanations
   - How the system works
   - Mental models and terminology
   - Limits and constraints

3. Reference material
   - API docs
   - Config options
   - Permissions and schemas
   - Error codes

4. Edge cases & troubleshooting
   - Common mistakes
   - Debugging steps
   - Known issues

5. Advanced usage
   - Power workflows
   - Integrations
   - Automation
   - Scaling considerations

6. Change history
   - Changelog
   - Deprecations
   - Migration guides

What should NOT be in the docs:
- Sales copy
- Value propositions
- Competitive comparisons
- Pricing explanations (unless feature-gated)


DECISION TEST
-------------
Ask: "Is the reader deciding, or doing?"

- Deciding → Website
- Doing    → Docs

Quick check:
- "Is this for me?" → Website
- "What does this do?" → Website
- "How do I set this up?" → Docs
- "Why isn't this working?" → Docs
- "Is this secure?" → Website (summary) + Docs (details)


HOW THEY WORK TOGETHER
----------------------
- Website makes the promise
- Docs fulfill the promise
- Website links into docs
- Docs avoid repeating marketing
- Terminology stays consistent
- Docs assume user intent

Short version:
Website = WHY
Docs     = HOW
